Configurar a estrutura completa do monorepo Azimute com pnpm workspaces e Turborepo, e scaffoldar os três pacotes com código funcional:
- `apps/web` — Next.js 15 + Tailwind v4 + Auth.js v5 + TanStack Query
- `apps/api` — Express.js v5 + Prisma 6 + ioredis + Socket.io v4
- `packages/shared` — tipos TypeScript e schemas Zod compartilhados

**Por que:** Esta tarefa é o fundamento de toda a Fase 1 do MVP. Sem o monorepo funcional, nenhum outro ticket de desenvolvimento pode ser iniciado. A Abordagem A (Scaffold Completo) foi escolhida para eliminar dívida técnica de setup e permitir que os próximos tickets comecem direto na lógica de negócio.

## Regras de Negócio

### Workspace Resolution

1. `apps/web` e `apps/api` referenciam `@azimute/shared` via workspace protocol: `"@azimute/shared": "workspace:*"`
2. Imports nos apps: `import { ... } from '@azimute/shared'`
3. Em dev mode, o TypeScript resolve `@azimute/shared` direto para `packages/shared/src/index.ts` via path no `tsconfig.json` (sem precisar de build do shared)

### Turborepo Pipeline

| Task | Cache | `dependsOn` | `persistent` |
|---|---|---|---|
| `build` | sim | `^build` | não |
| `dev` | não | — | sim |
| `lint` | sim | `^build` | não |
| `type-check` | sim | `^build` | não |
| `test` | não | `^build` | não |
| `test:watch` | não | — | sim |

### Validação de Ambiente (apps/api)

- `src/config/env.ts` usa Zod para validar **todas** as variáveis de ambiente na inicialização
- Se alguma variável obrigatória faltar, o processo encerra com mensagem de erro clara listando os campos inválidos
- O servidor nunca inicializa em estado parcialmente configurado

### Inicialização do Servidor (apps/api)

- Conexões com PostgreSQL e Redis são lazy (falha na primeira query, não no boot)
- A rota `GET /health` verifica a conectividade real do banco e do Redis; retorna `503` se indisponíveis
- Socket.io é inicializado no mesmo servidor HTTP (não em porta separada)

### Configuração de Portas

- `apps/api`: porta lida de `process.env.API_PORT` (padrão: `4000`)
- `apps/web`: porta padrão `3000` (configurada pelo Next.js)

### TypeScript Strict

- Todos os `tsconfig.json` usam `"strict": true`
- `any` é proibido em todo o projeto
- Alias `@/*` → `./src/*` em cada app

## Critérios de Aceite

**Dado** um ambiente com Node.js 22+, pnpm 9+ e Docker instalados,

**Quando** o desenvolvedor executa:
```bash
cp .env.example .env
docker compose -f docker-compose.dev.yml up -d
pnpm install
pnpm dev
```

**Então:**

1. `pnpm install` conclui sem erros, instalando dependências de todos os pacotes
2. `pnpm dev` sobe `apps/web` na porta 3000 e `apps/api` na porta 4000 simultaneamente
3. `GET http://localhost:4000/health` retorna HTTP 200 com `{ status: "ok", timestamp: "...", version: "0.1.0" }`
4. `GET http://localhost:3000` retorna a landing page com título "Azimute" sem erros no console
5. Modificar `apps/web/src/app/page.tsx` recarrega automaticamente (hot reload Next.js)
6. Modificar `apps/api/src/server.ts` reinicia automaticamente o servidor (tsx watch)
7. `pnpm build` compila os três pacotes sem erros TypeScript
8. `pnpm lint` passa sem erros em todos os pacotes
9. `pnpm type-check` passa sem erros em todos os pacotes
10. `pnpm test` executa sem erros (testes unitários mínimos do shared e api)

## Edge Cases

1. **`packages/shared` sem build durante dev** — resolver via `paths` no `tsconfig.json` de cada app apontando diretamente para `packages/shared/src/index.ts`; Turborepo não precisa buildar shared para dev funcionar

2. **Porta já em uso (EADDRINUSE)** — `apps/api` deve capturar o erro no evento `error` do servidor HTTP e exibir mensagem explícita com a porta conflitante, não um stack trace

3. **`.env` não configurado** — `env.ts` com Zod exibe todas as variáveis com erro agrupadas, com exemplo do valor esperado (do `.env.example`)

4. **PostgreSQL/Redis não acessíveis no boot** — a inicialização não trava; apenas a rota `/health` retorna `503` com `{ status: "degraded", postgres: false, redis: false }`

5. **`pnpm install --frozen-lockfile` em CI** — garantir que `pnpm-lock.yaml` seja gerado e comitado no primeiro `pnpm install`

6. **Node.js < 22** — `package.json` raiz define `"engines": { "node": ">=22", "pnpm": ">=9" }`; pnpm exibe aviso automático

7. **Importar `@azimute/shared` em testes Vitest sem build** — resolver via alias no `vitest.config.ts` de cada app apontando para `../../packages/shared/src/index.ts`

8. **Tailwind v4 + CSS não processado** — garantir que `postcss.config.mjs` com `@tailwindcss/postcss` está configurado; sem `tailwind.config.ts` (v4 é CSS-first)

## UI

### Landing Page (`apps/web/src/app/page.tsx`)

Página estática para confirmar que o app está rodando. Não possui estados de loading/error.

**Conteúdo:**
- Título: "Azimute"
- Subtítulo: "Quiz interativo para viagens escolares"
- Badge com o ambiente (`process.env.NODE_ENV`)
- Link `<a href="/login">` (rota futura, sem implementação)

**Estilização:** Tailwind v4 via classes utilitárias. Centralizado na tela com flexbox.

### Rota de Health (`GET /health`)

Não é uma UI, mas é o ponto de verificação mais importante do critério de aceite:

```json
{
  "status": "ok",
  "timestamp": "2026-05-12T00:00:00.000Z",
  "version": "0.1.0",
  "services": {
    "postgres": true,
    "redis": true
  }
}
```

Em caso de degradação:
```json
{
  "status": "degraded",
  "services": { "postgres": false, "redis": true }
}
```
HTTP 503 quando qualquer serviço estiver indisponível.
