## Modelo de Dados

### Modelos existentes (manter estrutura, revisar índices)

```typescript
// Tipos TypeScript derivados do schema (referência)
type Role = 'TEACHER' | 'ADMIN'
type SessionStatus = 'LOBBY' | 'ACTIVE' | 'QUESTION' | 'REVEAL' | 'LEADERBOARD' | 'FINISHED'
type MediaType = 'IMAGE' | 'VIDEO'
type OptionColor = 'RED' | 'BLUE' | 'YELLOW' | 'GREEN'

  id: string          // cuid

  email: string       // unique
  passwordHash: string
  role: Role
  createdAt: Date
  updatedAt: Date
}

  description?: string
  coverImage?: string

  authorId: string

  mediaType?: MediaType
  mediaUrl?: string

  id: string          // cuid
  questionId: string

  order: number

  id: string          // cuid
  quizId: string

}

  id: string          // cuid
  sessionId: string
  nickname: string

  joinedAt: Date
  // UNIQUE(sessionId, nickname)
}

  id: string          // cuid
  sessionId: string
  participantId: string

  optionId: string
  isCorrect: boolean

  answeredAt: Date
  // UNIQUE(participantId, questionId)

```

### Índices a adicionar (delta do schema atual)

| Modelo | Campo(s) | Tipo | Justificativa |
|---|---|---|---|
| `GameSession` | `status` | `@@index` | Filtro frequente ao buscar sessões ativas |
| `GameSession` | `pin` | já tem `@unique` | Nenhum índice adicional necessário |
| `GameParticipant` | `sessionId` | `@@index` | Query de todos os participantes de uma sessão |
| `GameAnswer` | `sessionId` | `@@index` | Leaderboard: busca todas respostas de uma sessão |
| `GameAnswer` | `questionId` | `@@index` | Reveal: busca todas respostas de uma questão |
| `Question` | `quizId, order` | `@@index` | Ordenação das questões ao carregar quiz |

## Regras de Negócio

1. **PIN único e imutável:** `GameSession.pin` é gerado externamente (lógica no service, não no banco). A constraint `@unique` no schema garante unicidade.
2. **Nickname único por sessão:** `@@unique([sessionId, nickname])` impede dois participantes com o mesmo nickname na mesma sessão.
3. **Uma resposta por questão por participante:** `@@unique([participantId, questionId])` em `GameAnswer`.
4. **Cascade deletes:** Questão deletada → `AnswerOption` deletados (`onDelete: Cascade`). Garantido pelo schema.
5. **Ordem das questões:** campo `order` em `Question` e `AnswerOption` define a sequência. Responsabilidade do service manter a integridade.
6. **Score inicial zero:** `GameParticipant.score` inicia em 0 e é incrementado via service ao processar `GameAnswer`.
7. **currentQuestionIndex:** Inicia em 0, avança via service. Não tem constraint de FK com `Question` — é responsabilidade do service validar o índice.

## Edge Cases

1. **Reexecução do seed:** Seed não deve falhar se os dados já existirem — usar `upsert` em vez de `create`.
2. **Schema com índices novos:** Ao rodar `prisma migrate dev` localmente pela primeira vez, todos os índices devem ser criados junto na migration inicial (não requerem segunda migration).
3. **DATABASE_URL ausente:** `prisma migrate dev` falhará com erro claro — comportamento esperado e correto, pois `env.ts` já valida isso.
4. **Opção `isCorrect: true` em múltiplas opções:** O schema permite (sem constraint). As regras de negócio garantem apenas 1 correta — responsabilidade do service/seed.
5. **Seed em ambiente de teste:** O arquivo `seed.ts` usa `DATABASE_URL` do env — em CI/test, deve apontar para um banco separado.
6. **Campos nullable vs optional:** `mediaType` e `mediaUrl` em `Question` são `?` (nullable) — o seed de exemplo cria questão sem mídia para cobrir o caso padrão.

Esta tarefa é de infraestrutura de banco. Os testes unitários de lógica de negócio pertencem às features que usam o schema — não a esta tarefa.

### O que testar nesta tarefa

**Testes de integração do seed** (arquivo `apps/api/prisma/__tests__/seed.test.ts`):

| Cenário | Tipo | O que validar |
|---|---|---|
| Seed cria professor | Integração | `User` com `role: TEACHER` existe após seed |
| Seed cria quiz publicado | Integração | `Quiz` com `isPublished: true` vinculado ao professor |
| Seed cria questões com opções | Integração | Cada questão tem 4 `AnswerOption`, exatamente 1 com `isCorrect: true` |
| Seed é idempotente | Integração | Rodar `seed` 2x não duplica dados nem lança erro |
| Constraints únicas | Integração | Tentar inserir participante com nickname duplicado na sessão lança erro `P2002` |
| Cascade delete | Integração | Deletar `Question` remove `AnswerOption` associados automaticamente |

**Estratégia de ambiente de teste:**
- Usar banco PostgreSQL dedicado para testes (variável `DATABASE_URL` com banco `azimute_test`)
- Setup: `beforeAll` aplica migrations + roda seed
- Teardown: `afterAll` limpa tabelas (ou usa transações revertidas)
- Conforme `.claude/rules/react-testing.md`: mockar apenas fronteira; aqui a fronteira é o `PrismaClient` — em testes de integração real, usar banco real (não mock)
