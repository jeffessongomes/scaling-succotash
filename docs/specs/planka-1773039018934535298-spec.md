## Objetivo

Expor 3 endpoints REST para o ciclo de vida de sessões de jogo:

1. **POST `/api/game-sessions`** — professor cria sessão para um quiz publicado; sistema gera PIN único de 6 dígitos
2. **GET `/api/game-sessions/:pin`** — aluno busca sessão pelo PIN para ingressar via WebSocket
3. **DELETE `/api/game-sessions/:id`** — professor encerra sessão antecipadamente por ID

O PIN é o identificador humano que alunos digitam; o ID é o identificador interno (cuid) para operações programáticas (ex: painel do professor).

```typescript
model GameSession {
```

## Regras de Negócio

### POST /api/game-sessions

| # | Regra |
|---|---|
| RN-01 | Quiz deve existir |
| RN-02 | Quiz deve estar publicado (`isPublished: true`) |
| RN-03 | Quiz deve ter pelo menos 1 pergunta |
| RN-04 | Não pode existir sessão não-finalizada (`status != FINISHED`) para o mesmo quiz |
| RN-05 | PIN gerado com 6 dígitos numéricos, único via verificação no Redis |
| RN-06 | Sessão persiste no PostgreSQL com `status: LOBBY` |
| RN-07 | Estado inicial é salvo no Redis com TTL de 4h |
| RN-08 | Requer autenticação de professor |

### GET /api/game-sessions/:pin

| # | Regra |
|---|---|
| RN-09 | PIN deve existir no Redis ou no PostgreSQL |
| RN-10 | **Sessão só é retornada se status for LOBBY ou ACTIVE** (aluno não pode entrar em sessão finalizada/em andamento avançado) |
| RN-11 | Não requer autenticação (aluno acessa diretamente) |
| RN-12 | Não expõe `hostSocketId` nem `hostDisconnectedAt` na resposta |
| RN-13 | Resposta inclui `participantCount` (não os objetos de participantes) |

### DELETE /api/game-sessions/:id

| # | Regra |
|---|---|
| RN-14 | Busca sessão por `sessionId` (cuid) — primeiro no Redis via scan/Postgres, não por PIN |
| RN-15 | Sessão deve existir |
| RN-16 | Apenas o professor que criou a sessão (`authorId`) pode encerrá-la |
| RN-17 | Persiste dados no PostgreSQL (`status: FINISHED`, `endedAt`) e remove do Redis |
| RN-18 | Requer autenticação de professor |

### Limite de Participantes (transversal)

| # | Regra |
|---|---|
| RN-19 | Máximo de 60 participantes por sessão |
| RN-20 | A verificação de 60 participantes ocorre no `player:join` WebSocket (não neste REST endpoint), mas a constante `MAX_PARTICIPANTS = 60` deve ser exportada de `game.types.ts` para uso compartilhado |

## Critérios de Aceite

### POST /api/game-sessions

```
Dado que o professor está autenticado
E o quiz existe, está publicado e tem pelo menos 1 pergunta
E não há sessão ativa para esse quiz
Quando POST /api/game-sessions com { quizId }
Então retorna 201 com { sessionId, pin (6 dígitos), status: 'LOBBY', quizTitle, totalQuestions }
E o pin é único
E a sessão persiste no PostgreSQL com status LOBBY
E a sessão é salva no Redis com TTL 4h

Dado que o quiz não está publicado
Quando POST /api/game-sessions
Então retorna 422

Dado que já existe sessão ativa para o quiz
Quando POST /api/game-sessions
Então retorna 409

Dado que o quiz não existe
Quando POST /api/game-sessions
Então retorna 404

Dado que o professor não está autenticado
Quando POST /api/game-sessions
Então retorna 401
```

### GET /api/game-sessions/:pin

```
Dado que a sessão existe e está em status LOBBY ou ACTIVE
Quando GET /api/game-sessions/:pin (sem autenticação)
Então retorna 200 com { sessionId, pin, status, currentQuestionIndex, quizId, participantCount }
E a resposta NÃO contém hostSocketId nem hostDisconnectedAt

Dado que a sessão existe mas está em status FINISHED, QUESTION, REVEAL ou LEADERBOARD
Quando GET /api/game-sessions/:pin
Então retorna 404 (PIN não disponível para ingresso)

Dado que o PIN não existe
Quando GET /api/game-sessions/:pin
Então retorna 404
```

### DELETE /api/game-sessions/:id

```
Dado que o professor está autenticado e é o criador da sessão
E a sessão existe
Quando DELETE /api/game-sessions/:id
Então retorna 204
E a sessão é finalizada no PostgreSQL (status: FINISHED, endedAt definido)
E os dados do Redis são removidos

Dado que o professor autenticado não é o criador da sessão
Quando DELETE /api/game-sessions/:id
Então retorna 403

Dado que a sessão não existe
Quando DELETE /api/game-sessions/:id
Então retorna 404

Dado que o professor não está autenticado
Quando DELETE /api/game-sessions/:id
Então retorna 401
```

## Edge Cases

1. **PIN collision**: a função `generateUniquePin` tenta até 10 vezes → em caso de falha lança `Error`; o `errorHandler` deve mapear para 500
2. **Sessão no PostgreSQL mas não no Redis** (TTL expirado): `getSessionByPin` já lida com fallback para DB, mas a sessão de Redis estará desatualizada — lógica de fallback já existente é suficiente
3. **DELETE /api/game-sessions/:id com sessão em status ACTIVE**: permitido — professor pode encerrar a qualquer momento; `finalizeSession` persiste dados parciais
4. **Dois PINs iguais em geração concorrente**: a verificação `redis.exists()` no `generateUniquePin` é atômica o suficiente para ambiente single-instance; documentar limitação para multi-instância
5. **Quiz sem perguntas**: `RN-03` — retorna 422 com mensagem clara
6. **GET com PIN de sessão QUESTION/REVEAL/LEADERBOARD**: retorna 404 (aluno não pode reingressar via PIN em sessão em andamento; a reconexão é via WebSocket com o mesmo nickname)

```typescript
export async function getSessionById(id: string): Promise<GameSessionState | null> {
  // Estratégia: busca no DB, pega o pin, busca no Redis
  const dbSession = await prisma.gameSession.findUnique({
    where: { id },
    select: { pin: true, ... }
  })
  if (!dbSession) return null
  return getSessionByPin(dbSession.pin)
}
```
