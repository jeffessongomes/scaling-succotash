# Spec: [Bug] optionId descartado em player:answer — distribuição sempre zerada

**ticket_id:** `planka-1773228224952665487`
**Branch:** `feat/planka-1773039018817094784-backend-engine-jogo` (fix aplicado na mesma branch do PR #7)
**Descoberto em:** Review do PR #7

---

## Objetivo

Corrigir três bugs encadeados causados pelo descarte do `optionId` no handler `player:answer`:

1. **`saveAnswer` não persiste `optionId`** — o Redis armazena apenas `answeredInMs` como valor do hash.
2. **`revealAnswers` produz distribuição zerada** — usa `String(answeredInMs)` como chave, que nunca bate com IDs de opção.
3. **`isCorrect` sempre verdadeiro** — a verificação checa apenas `!== undefined`, não compara com `correctOptionId`.

O bug também contamina `finalizeSession` em `game.service.ts`, que grava registros com `optionId` errado e `isCorrect` incorreto no banco.

---

## Modelo de Dados

### Contrato Redis (antes × depois)

| | Antes | Depois |
|---|---|---|
| **Chave** | `game:answers:<pin>:<questionId>` | `game:answers:<pin>:<questionId>` |
| **Campo** | `participantId` | `participantId` |
| **Valor** | `"5000"` (answeredInMs) | `"opt-correct:5000"` (`optionId:answeredInMs`) |

### Tipo `AnswerRecord` (novo tipo compartilhado)

```typescript
export interface AnswerRecord {
  optionId: string
  answeredInMs: number
}
```

### Assinaturas alteradas

```typescript

saveAnswer(pin, questionId, participantId, optionId, answeredInMs): Promise<boolean>
getAnswers(pin, questionId): Promise<Record<string, AnswerRecord>>
```

## Regras de Negócio

1. `saveAnswer` serializa o valor como `"${optionId}:${answeredInMs}"` antes de chamar `HSETNX`.
2. `getAnswers` parseia o valor dividindo pelo primeiro `:` — `optionId` pode conter `:` no futuro (IDs UUID não contêm, mas por segurança usar `split(':')` com `limit 2` ou `indexOf`).
3. Em `revealAnswers` (socket):
   - `isCorrect = answer.optionId === correctOptionId`
   - `distribution[answer.optionId]++` — só incrementa se `answer.optionId` estiver nas chaves de `distribution`
4. Em `finalizeSession` (service):
   - `optionId = answer.optionId`
   - `isCorrect = answer.optionId === correctOption?.id`
   - `answeredInMs = answer.answeredInMs`
5. O campo `answeredCount` em `session:answer-received` continua usando `Object.keys(getAnswers(...)).length` — nenhuma mudança de comportamento.

## Critérios de Aceite

### Dado que um jogador responde corretamente

**Quando** o host revela respostas,  
**Então** `session:answers-revealed.distribution[correctOptionId]` é `1` e `isCorrect` é `true` para o jogador na persistência.

### Dado que um jogador responde incorretamente

**Quando** o host revela respostas,  
**Então** `session:answers-revealed.distribution[wrongOptionId]` é `1`, `distribution[correctOptionId]` é `0`, e `isCorrect` é `false` na persistência.

### Dado que dois jogadores respondem opções diferentes

**Quando** o host revela respostas,  
**Então** `distribution` soma corretamente por opção (ex: `{ 'opt-a': 1, 'opt-b': 1, 'opt-c': 0 }`).

### Dado que nenhum jogador respondeu

**Quando** o host revela respostas,  
**Então** `distribution` é `{ [optId]: 0 }` para todas as opções.

### Dado que `saveAnswer` é chamado duas vezes para o mesmo participante

**Quando** a segunda chamada ocorre,  
**Então** retorna `false` e o Redis não é sobrescrito (HSETNX garante idempotência).

---

## Edge Cases

1. **`optionId` inválido (não pertence à pergunta):** `distribution` não incrementa (chave inexistente em `distribution`). `isCorrect` resulta `false`.
2. **Tempo expirado (`answeredInMs > timeLimitMs`):** já bloqueado antes de `saveAnswer` — sem mudança.
3. **Redis indisponível:** `saveAnswer` lança exceção — sem mudança no comportamento de erro.
4. **`optionId` contendo `:`:** usar `value.split(':')` com índice fixo — `optionId = parts[0]`, `answeredInMs = Number(parts.slice(1).join(':'))`. Como UUIDs e IDs do Prisma não contêm `:`, o split simples no primeiro `:` via `indexOf` é suficiente e mais robusto.
5. **`getAnswers` com hash vazio (nenhuma resposta):** `hgetall` retorna `{}` — retorna `Record<string, AnswerRecord>` vazio, sem erro.
6. **`finalizeSession` chamada em sessão sem respostas:** loop não itera, `answerRecords` permanece `[]`.

#### `game-state.cache.ts`
```typescript
// saveAnswer — novo parâmetro optionId
export async function saveAnswer(
  pin: string,
  questionId: string,
  participantId: string,
  optionId: string,       // novo
  answeredInMs: number,
): Promise<boolean> {
  const result = await redis.hsetnx(
    `game:answers:${pin}:${questionId}`,
    participantId,
    `${optionId}:${answeredInMs}`,  // serialização
  )
  return result === 1
}

// getAnswers — retorno Record<string, AnswerRecord>
export async function getAnswers(
  pin: string,
  questionId: string,
): Promise<Record<string, AnswerRecord>> {
  const raw = await redis.hgetall(`game:answers:${pin}:${questionId}`)
  const result: Record<string, AnswerRecord> = {}
  for (const [participantId, value] of Object.entries(raw)) {
    const colonIdx = value.indexOf(':')
    result[participantId] = {
      optionId: value.slice(0, colonIdx),
      answeredInMs: Number(value.slice(colonIdx + 1)),
    }
  }
  return result
}
```

  const participant = session.participants[participantId]
