## Regras de Negócio

1. O `socket.ts` deve criar **uma única instância** do socket (singleton) — não conectar automaticamente na importação (conectar via `socket.connect()` explícito nas features).
2. O `api-client.ts` existente usa `axios`. Manter e adicionar interceptor de erro padronizado (se ausente).
3. O `ui.store.ts` não deve persistir estado entre refreshes (sem `localStorage` neste ticket).
4. O `tailwind.config.ts` deve usar tokens do tema Azimute (cores primárias gamificadas — roxo/azul vibrante), mantendo compatibilidade com shadcn/ui.
5. A estrutura de `features/` deve conter apenas arquivos `index.ts` de barrel e diretórios vazios — **sem implementação de negócio** neste ticket.

## Edge Cases

1. **Socket conectado antes do tempo** — `autoConnect: false` garante que o socket não tenta conectar ao importar o módulo; só conecta quando a feature de jogo chamar `socket.connect()`.
2. **Múltiplas instâncias de QueryClient em SSR** — `query-client.ts` já trata isso com o padrão de singleton no browser. O `providers.tsx` deve chamar `getQueryClient()`, não `makeQueryClient()` diretamente (já está correto).
3. **Conflito de versão Tailwind + shadcn** — shadcn/ui suporta Tailwind v4 a partir da versão `shadcn@latest` (2.x). Usar `pnpm dlx shadcn@latest init` (sem `@canary`).
4. **Diretórios vazios no git** — criar `.gitkeep` nos diretórios que são apenas estrutura, sem arquivos de código ainda.
5. **Path alias `@/`** — já configurado em `tsconfig.json` e `vitest.config.ts`. O `components.json` do shadcn deve usar `@/components` para manter consistência.

```typescript
import { useEffect, useState } from 'react'

export function useMediaQuery(query: string): boolean {
  const [matches, setMatches] = useState(false)

  useEffect(() => {
    const media = window.matchMedia(query)
    setMatches(media.matches)
    const listener = (e: MediaQueryListEvent) => setMatches(e.matches)
    media.addEventListener('change', listener)
    return () => media.removeEventListener('change', listener)
  }, [query])

  return matches
}
```

## Abordagens Propostas

### Abordagem A — shadcn CLI + scaffold manual (recomendada)

1. Executar `pnpm dlx shadcn@latest init` em `apps/web` (interativo: New York style, paleta Slate como base, path alias `@/`)
2. O CLI gera `components.json` e `tailwind.config.ts` automaticamente
3. Complementar `tailwind.config.ts` com tokens do tema Azimute manualmente
4. Criar todos os arquivos de scaffold (`socket.ts`, `ui.store.ts`, etc.) manualmente

**Pro:** setup shadcn idiomático, CSS variables geradas corretamente.
**Con:** requer interação com o CLI (pode ser não-interativo com flags).

### Abordagem B — Setup totalmente manual

1. Criar `components.json` manualmente com a config do shadcn/ui
2. Criar `tailwind.config.ts` com tema completo
3. Criar todos os arquivos de scaffold

**Pro:** controle total, sem dependência de CLI interativo.
**Con:** mais trabalhoso; risco de erro na config do shadcn.

### Abordagem C — Mínimo viável

1. Apenas `lib/socket.ts` + estrutura de pastas + stores/types
2. Adiar shadcn/ui para quando a primeira feature precisar de um componente UI

**Pro:** mais rápido, sem risco de conflito de versão.
**Con:** shadcn/ui estará faltando para quem for implementar as features de UI logo depois.
