## Análise Qualitativa

### Aderência à spec

**Implementado corretamente:**
- Estrutura de arquivos segue exatamente o definido na spec
- Contratos de API consumidos corretos (todos os endpoints mapeados)
- Reorder otimista implementado conforme pseudocódigo da spec
- Edge cases de publicação (0 perguntas), lista vazia e reorder com falha cobertas
- DnD com drag handle ocultado quando há 1 única pergunta (`[id]/page.tsx:247`)
- `quizKeys` centralizado em `query-keys.ts` conforme padrão definido

**Gaps identificados:**
1. **`QuizEditorPage` sem testes** — A spec define explicitamente 5 casos de teste para `QuizEditorPage.test.tsx` e nenhum foi implementado. O arquivo `/quizzes/[id]/page.tsx` é um orquestrador (une múltiplos domínios: quiz detail, DnD, form de pergunta, form de opção, ações de publicação). Pelas regras do projeto, **orquestradores devem ser testados**.
2. **CA6 sem teste positivo** — O cenário "when delete is confirmed → quiz is removed from list" está ausente. Apenas o cenário de cancelamento foi testado.
3. **CA15 comportamento divergente** — Spec define "desmarcação automática" da opção correta anterior; implementação desabilita o checkbox de novas opções. Usuário que errou ao marcar uma opção como correta não consegue corrigir sem deletar a opção.
4. **Dashboard sem testes** — `dashboard/page.tsx` não tem arquivo de teste correspondente (spec não define explicitamente, mas a página é uma agregação de dados que pode ter regressões silenciosas).

### Qualidade

O código é limpo, bem estruturado e segue boas práticas:
- Separação clara entre camadas (API → hooks → páginas/componentes)
- Nenhum uso de `any` em todo o código-fonte
- `useQuizEditor` encapsula corretamente a lógica otimista de reorder
- `ConfirmDialog` como componente genérico reutilizável é uma boa decisão
- `mocks/quiz.ts` com factories tipadas e wrappers semânticos por cenário

### Sugestões (não bloqueantes)

- O botão de publicar no editor (`[id]/page.tsx:216`) usa `publish.mutate` (fire-and-forget), enquanto a lista de quizzes usa `publish.mutateAsync`. Seria mais consistente usar o mesmo padrão em ambos.
- `AnswerOptionForm` usa `useWatch` para detectar a cor selecionada mas o campo `color` é hidden no form. Poderia usar `setValue` + `getValues` sem `useWatch` para evitar re-renders desnecessários.
