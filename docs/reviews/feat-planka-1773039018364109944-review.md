## Análise Qualitativa

### Aderência à Spec

O PR corresponde com fidelidade ao definido na spec. Todos os arquivos previstos foram criados (`auth.schema.ts`, `auth.service.ts`, `auth.router.ts`, `__tests__/auth.router.test.ts`) e os modificados foram ajustados conforme esperado (`app.ts`, `auth.ts`). O middleware, a extensão de tipos e a constante `ROUTES` estão todos no lugar.

Destaque positivo: a mensagem genérica `"Credenciais inválidas"` é retornada tanto para email inexistente quanto para senha incorreta, implementando corretamente a prevenção de enumeração de emails definida na regra de negócio 3.

Ponto de atenção (não-bloqueante): o `authorize()` no `auth.ts` não tem teste unitário isolado no web. A spec prevê isso e delega para E2E (`/proj-e2e`). Aceitável para este ticket.

### Qualidade

- **Segurança:** Argon2id com `timeCost: 3, memoryCost: 65536` — parâmetros corretos conforme spec
- **Erros HTTP:** semântica correta: 201 (registro), 200 (login), 400 (validação Zod), 401 (credenciais), 409 (conflito)
- **Sem vazamento de `passwordHash`:** verificado nos testes (`response.body.passwordHash` → `undefined`)
- **Edge cases cobertos:** body inválido, senha curta, email malformado — todos com testes

### Sugestões (não-bloqueantes)

1. **Factory function nos testes de router:** os mocks do Prisma são criados inline com `as unknown as User` (ex: `auth.router.test.ts:38-46`). A convenção do projeto recomenda factory functions tipadas para cenários reutilizados. Com 8 testes e dois perfis distintos (TEACHER / ADMIN), uma `createUserFixture()` evitaria repetição e removeria o `as unknown as User`. Não é bloqueante — o padrão atual funciona e os tipos estão corretos na prática.
2. **Matcher do middleware:** `/quizzes/:path*` está no matcher, mas a spec menciona apenas `/dashboard`. Verificar se essa proteção adicional foi intencional. Não é erro — pode ter sido adicionado antecipando o ticket de quizzes.
